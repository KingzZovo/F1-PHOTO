## 1. Real ONNX inference pipeline

### Current state

- `libonnxruntime.so` (1.20.1, aarch64) loads cleanly and is reported via
  `GET /api/admin/models` (`ort_available=true`, `ready=true`).
- All five model slots (`face_detect`, `face_embed`, `object_detect`,
  `generic_embed`, `angle_classify`) load successfully — but only because
  `models/*.onnx` are **stub Identity graphs** generated by
  `packaging/scripts/make-stub-models.py` (~145 bytes each). They are not
  SCRFD / ArcFace / YOLOv8n / DINOv2-small / MobileNetV3 weights.
- `worker::run_real_pipeline` is a **placeholder** that immediately returns
  `Err("real ONNX inference pipeline not yet wired ...")`. The detection,
  embedding, persistence, and pgvector-recall code paths are sketched in
  comments but not implemented.
- Runtime behaviour is gated on `F1P_INFERENCE_STUB_FALLBACK` (default
  **on**). When the env flag is on, a real-pipeline `Err` is downgraded to
  `Outcome::FallbackUnmatched` so the queue drains, photos transition to
  `unmatched`, and the API stays usable. Flip to `0` once the real
  pipeline is wired so genuine errors surface as queue retries again.
- `recognition_items` therefore stays empty. The recall buckets
  (matched / learning / unmatched) and the gallery-augmentation step at
  `score >= 0.95` are **not exercised** in the current smoke run.

### What "done" looks like

1. Replace `models/*.onnx` stubs with real exported weights:
   - SCRFD 500m or 2.5g (face detect)
   - ArcFace MobileFaceNet 512d (face embed)
   - YOLOv8n trained on a tool/device class set (object detect)
   - DINOv2-small (generic embed)
   - MobileNetV3 angle classifier (optional)
   Each must match the input/output shape conventions used by
   `inference::preprocess` (NCHW float32, SCRFD/ArcFace norms, YOLO
   letterbox at 640).
2. Implement `worker::run_real_pipeline`:
   1. Decode photo via the `image` crate.
   2. Letterbox + normalise per `preprocess::Norm`.
   3. `state.models.get(ModelKind::FaceDetect)?.run(inputs! { "input" => arr })?`
      → bboxes; crop; embed via `FaceEmbed`.
   4. Same for `ObjectDetect` + `GenericEmbed`.
   5. INSERT `detections` rows with `recall::encode_vector(&emb)`.
   6. pgvector cosine recall via the helpers in
      `inference::recall` against `Thresholds::DEFAULT` (or per-project).
   7. Bucket each detection (matched / learning / unmatched), INSERT
      `recognition_items`, append to the gallery for `score >= 0.95`.
   8. Aggregate per-photo outcome and update `photos.status`.
3. Flip `F1P_INFERENCE_STUB_FALLBACK=0` in production deployment so
   real-pipeline errors surface as queue retries (and eventually photo
   `failed` after `MAX_ATTEMPTS=5`).
4. Strengthen `packaging/scripts/smoke-e2e.sh` to assert
   `recognition_items.total > 0` for the seeded happy-path photo (currently
   only asserts the endpoint returns 200 + that the photo leaves
   `processing`).
5. Add an integration test that uses the bundled-pg + a checked-in
   small-but-real face crop to verify at least one `recognition_items` row
   is produced.

### Why deferred

Real model weights are not in the repo (size + licensing) and require an
out-of-band fetch step. Wiring the inference pipeline against stub Identity
models would either (a) produce meaningless detection output that doesn't
match downstream shape expectations or (b) require shape-shimming logic
that would have to be torn out the moment real weights arrive. Treating
this as one cohesive "swap in real models" milestone — rather than half-
wiring against stubs — is cleaner.
