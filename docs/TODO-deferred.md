## 1. Real ONNX inference pipeline

### Current state (after this turn)

Partial progress: **the GenericEmbed slot is now wired to a real ONNX
model end-to-end.** The other three production model slots still run
against stub Identity graphs.

- `libonnxruntime.so` (1.20.1, aarch64) loads cleanly and is reported via
  `GET /api/admin/models` (`ort_available=true`, `ready=true`).
- `models/generic_embed.onnx` is now a **real DINOv2-small ONNX export**
  (88,459,888 bytes; md5 `da9c90b7f0b46c9287d357d3be273236`; opset 11;
  input `pixel_values` shape `(B,3,H,W)` f32; output `last_hidden_state`
  shape `(B,257,384)` f32). Source:
  `https://huggingface.co/Xenova/dinov2-small/resolve/main/onnx/model.onnx`.
- The other four slots (`face_detect.onnx`, `face_embed.onnx`,
  `object_detect.onnx`, `angle_classify.onnx`) are still **stub Identity
  graphs** (~145 bytes each) generated by
  `packaging/scripts/make-stub-models.py`. SCRFD / ArcFace / YOLOv8n /
  MobileNetV3 weights still need to be sourced (HuggingFace insightface
  mirror returned 502 transient at the time of this turn; ultralytics
  `.pt` → ONNX export is the planned alternate route).
- `worker::run_real_pipeline` now actually runs:
  1. Decode + 224×224 letterbox + ImageNet normalisation.
  2. DINOv2-small `session.run` with `pixel_values`.
  3. Extract the CLS token from `last_hidden_state[0, 0, :]` (384 f32s).
  4. L2-normalise; zero-pad 384 → 512 to match the schema's
     `vector(512)` width (cosine similarity is preserved when both
     compared sides share the zero-pad pattern, which holds for the
     single-model gallery).
  5. INSERT one `detections` row (`target_type='tool'`, bbox = full
     image, score = 1.0; this is a placeholder for a real per-object
     detection step which arrives with YOLOv8n in §2 below).
  6. pgvector cosine recall against the tool/device gallery via
     `recall::top1_object` and `Thresholds::DEFAULT` (`{0.50, 0.62,
     0.95}`).
  7. UPDATE the detection row with the bucketed match status + matched
     owner; INSERT a `recognition_items` row.
  8. On `score >= 0.95`, call `recall::augment` to grow the gallery.
- `F1P_INFERENCE_STUB_FALLBACK` still defaults **on** in the smoke env
  so any future regression in the real path stays visible (queue drains,
  photo gets `unmatched`) without blocking the API. Production should
  flip it off once all four ONNX slots are wired so real-pipeline `Err`s
  surface as queue retries.
- `packaging/scripts/smoke-e2e.sh` now asserts
  `recognition_items.total >= 1` for the uploaded happy-path photo. The
  fallback path does NOT write `recognition_items` rows, so this
  assertion catches the case where the real pipeline silently errors
  out and falls back to `unmatched` — i.e. it implements the
  "succeeded ≠ healthy" rule for this code path.
- The `Outcome::FallbackUnmatched` variant and the documented WARN line
  it logs ("real ONNX pipeline unavailable; falling back to unmatched")
  are still kept around for the three not-yet-wired slots. Once those
  ship, the WARN whitelist in `smoke-e2e.sh` should be tightened to
  reject that line entirely.

### What "done" looks like

1. Source real weights for the remaining three slots:
   - SCRFD 500m or 2.5g → `models/face_detect.onnx`.
   - ArcFace MobileFaceNet 512d → `models/face_embed.onnx`.
   - YOLOv8n trained on a tool/device class set →
     `models/object_detect.onnx`.
   - (Optional) MobileNetV3 angle classifier →
     `models/angle_classify.onnx`.
   Each must match the input/output shape conventions used by
   `inference::preprocess` (NCHW float32, SCRFD/ArcFace norms, YOLO
   letterbox at 640).
2. Extend `worker::run_real_pipeline`:
   - Run SCRFD on the whole image to get face bboxes; crop; run ArcFace
     on each crop to get a 512-d face embedding (no padding needed); for
     each crop, INSERT a `detections` row with `target_type='face'` and
     bucket against the `person` gallery.
   - Run YOLOv8n on the whole image to get tool/device bboxes; for each
     crop, run DINOv2 (the path that already works in this turn) and
     bucket against the tool/device gallery instead of the synthetic
     full-image embedding currently used.
   - Drop the synthetic full-image `target_type='tool'` row once YOLO is
     in place — it's a stand-in until then.
3. Flip `F1P_INFERENCE_STUB_FALLBACK=0` in production deployment so
   real-pipeline errors surface as queue retries (and eventually photo
   `failed` after `MAX_ATTEMPTS=5`). Tighten the smoke WARN whitelist to
   reject the fallback line entirely once all four slots are wired.
4. Add an integration test that uses the bundled-pg + a checked-in
   small-but-real face crop to verify at least one matched-bucket
   `recognition_items` row is produced (the smoke assertion currently
   covers the *unmatched* bucket on a freshly-seeded gallery).

### Why deferred

Real model weights are not in the repo (size + licensing) and require an
out-of-band fetch step. DINOv2-small was sourced from the HuggingFace
Xenova mirror this turn; SCRFD / ArcFace / YOLOv8n still need their
respective fetch + export pipelines. The DINOv2 path was wired first
because it is the simplest end-to-end smoke target (no detection
post-processing, single output tensor, single gallery) and proves the
ONNX → preprocess → session.run → pgvector chain works against the live
`vector(512)` schema before the heavier face / tool detection stacks
land.
