- `models/generic_embed.onnx` is now a **real DINOv2-small ONNX export**

  input `pixel_values` shape `(B,3,H,W)` f32; output `last_hidden_state`

  `https://huggingface.co/Xenova/dinov2-small/resolve/main/onnx/model.onnx`.
- `models/face_detect.onnx` is now a **real SCRFD det_500m** export
  from the buffalo_s pack (2,524,817 bytes; md5
  `51dc6ebafbd47fcd25b27b6e136062c8`; input `input.1 (1,3,?,?)` f32;
  9 outputs head-then-stride: scores @ s8/s16/s32 (idx 0..2), bboxes
  @ s8/s16/s32 (idx 3..5), kps @ s8/s16/s32 (idx 6..8); 2 anchors per
  cell). Confidence threshold 0.5, NMS IoU 0.4.
- `models/face_embed.onnx` is now a **real ArcFace MobileFaceNet 512d**
  export (`w600k_mbf`, 13,616,099 bytes; md5
  `c5b029527cb6f874e057613fc10bae1b`; input `input.1 (N,3,112,112)` f32;
  output `516 (1,512)` f32; opset 11). 512-d output — NO zero padding.
- `models/object_detect.onnx` is now a **real YOLOv8n COCO ONNX export**
  (12,824,004 bytes; md5 `564f3a7ee79c3302e4845b674efd538b`; opset 12;
  input `images (1,3,640,640)` f32; output `output0 (1,84,8400)` f32).
  Exported from `yolov8n.pt` (Ultralytics v8.3.0) via
  `YOLO("yolov8n.pt").export(format="onnx", opset=12, imgsz=640)`. The
  84 channels are `[cx, cy, w, h, c0..c79]` (post-sigmoid class probs);
  decode + class-agnostic NMS live in `inference::yolov8` with default
  `conf=0.25`, `iou=0.45`, `max_detections=10`.
- The `angle_classify.onnx` slot is still a **stub Identity graph** (~147
  bytes) generated by `packaging/scripts/make-stub-models.py`. It is
  **not** on the critical path; the recognition pipeline never invokes
  it. A real MobileNetV3 angle classifier remains optional future work.
- `worker::run_real_pipeline` now actually runs:
  1. Decode the photo once; reuse for both face and tool sub-pipelines.
  2. **Face pipeline (`run_face_pipeline`)**: SCRFD letterbox at 640
     + `Norm::Scrfd`, decode 9-output head into bboxes + per-face kps,
     112×112 ArcFace crop per face + `Norm::ArcFace`, ArcFace
     `session.run` for a 512-d L2-normalised embedding (no padding).
     INSERT one `detections` row per face (`target_type='face'`,
     `source="scrfd"`), recall against the person gallery via
     `recall::top1_face`, INSERT a `recognition_items` row, and
     `recall::augment` when `score >= 0.95`.
  3. **Tool/device pipeline (`run_tool_pipeline`)**: YOLOv8 letterbox at
     640 + `Norm::Unit`, decode `output0` into per-anchor xyxy boxes,
     class-agnostic NMS, top-K crops; per crop runs DINOv2-small at
     224×224 + `Norm::ImageNet`, takes the CLS token (384 f32s),
     L2-normalises, zero-pads to 512 via `recall::pad_to_512` (cosine
     preserved under shared zero-pad pattern), INSERTs one `detections`
     row per crop (`target_type='tool'`, `source="yolov8"`,
     `class_id=<COCO id>`), recalls against the tool/device gallery via
     `recall::top1_object`, INSERTs a `recognition_items` row, and
     augments on `score >= 0.95`. When YOLOv8 proposes zero boxes the
     pipeline falls back to a single whole-image DINOv2 detection
     (`source="dinov2_whole_image_fallback"`) so the smoke assertion
     `recognition_items.total >= 1` continues to hold.
  4. Per-photo outcome aggregates the best bucket across all detections
     (face + tool): Matched > Learning > Unmatched.
- `F1P_INFERENCE_STUB_FALLBACK` and the `stub_fallback_enabled()` helper
  have been **removed** as of milestone #1c. The cut-over chain was:
  Step C flipped the default to OFF, milestone #1 (#1a + #1b, commit
  1827b56) dropped the env export from smoke and confirmed no
  in-tree manifest sets it, and milestone #1c finally deleted the
  worker-side branch outright now that the real-dataset baselines
  (#2 face slice + #2-tool tool/device slice) have landed and we are

  at runtime.

  `recognition_items.total >= 1` for the uploaded happy-path photo. The
  fallback path does NOT write `recognition_items` rows, so this

- `packaging/scripts/smoke-e2e.sh` no longer whitelists the

  Any WARN line in the server log now fails the smoke, so a silent

### Important caveat: yolov8n is COCO-trained, not tool/device-specific

`yolov8n.pt` from ultralytics is trained on the 80-class COCO dataset.
Its class labels are unrelated to F1-photo's `tools` / `devices`
taxonomy: a wrench, a multimeter, or a project-specific instrument is
not a COCO category. Production-grade tool/device detection therefore
requires fine-tuning a domain-specific detector against a curated
tool/device crop dataset (or training from scratch with the project's
actual SKU set). For now, the pipeline uses YOLOv8 strictly as a
**class-agnostic region proposer**: any high-confidence object box gets
cropped, embedded with DINOv2, and re-identified via per-project
gallery cosine recall — which is the part of the system that actually
knows about the project's taxonomy. The COCO `class_id` is recorded on
each `detections` row's bbox JSON for forensics but does NOT influence
matching.

   is bottlenecked on the gallery quality, not the detector.

  `photos.owner_id`. When `owner_type='person'` the new
  `run_face_bootstrap_pipeline` runs SCRFD detect + ArcFace embed and,

  could never fire on later wo_raw uploads of the same person.

  seed nothing. The upload itself still succeeds (the API contract is
  unchanged) — it just doesn't produce a usable seed; the operator can
  re-shoot or correct.
- All detected faces in a single owner-known upload are attributed to
  the same owner. This matches the upload's intent ("this is person X"
  registration shots are expected to contain X's face) and is the v1
  simplification we want; multi-face disambiguation (largest-face-only,
  or reject-multi-face) can be added later if the false-seed rate
