real ONNX inference pipeline locked in by default (v1.5.0). It is the
authoritative status board for Steps A / B / C and the follow-on milestones.

## 2026-04-27 design decisions (King turn re #5 path)

- **Drop `device` as a distinct class for v1.5.0.** Both `wo_raw` and bootstrap paths write `target_type='tool'` only. The `detect_target` and `owner_type` ENUMs keep `device` to preserve historical rows; no new code emits it. Rationale: King has no business need to distinguish hand-tools vs in-place equipment in v1.5.0; collapsing the class avoids a multi-head detector + matching DB / API contract churn that the data does not justify.
- **Active learning is the centre of #5, not a curated training set.** King's standing requirement (越用越准) means the model must improve from operator corrections in production. The `#5` row is split into `#5-skel` (decouple worker hardcode + corrections schema) → `#7` (active-learning loop) → `#5-bootstrap` (first fine-tuned weights). No upfront 500-image labelled dataset is needed; the system bootstraps on COCO YOLOv8n and tightens with use.
- **Stay on YOLOv8n as the detector backbone.** CPU-friendly, ONNX path proven, ultralytics retrain tooling mature. RT-DETR / YOLOv8s rejected: too slow on aarch64 CPU and the data ceiling is so low that more capacity over-fits before it generalises.
- **Eastern-face P/R (`#2c-asia`) does not block v1.5.0.** Operator base is decided per-deployment; `#2c-asia` ships as a follow-up and can run in parallel with `#5-skel` / `#7` since the two areas touch disjoint code (SCRFD + face fixture vs YOLOv8n + tool corrections schema).
- **Acceptance numbers (plain-language).** `#5-bootstrap` ships when: the system reports ≤ 1.5 candidate items per photo (down from 3.26); when it says "this is a tool" it's right ≥ 85 % of the time; when there's actually a tool in the frame, it spots it ≥ 70 % of the time; face recognition does not get worse than today (Western F1 ≥ 0.667); cold-start does not slow more than 20 %.
- **#5-skel scope revision (discovered while implementing, 2026-04-27 +08).** The original `#5-skel` plan (recorded at c853773) had three items that turned out to be redundant with already-shipped code: (1) the planned `recognition_corrections` table is dominated by the existing `recognition_items.corrected_owner_{type,id,by,at}` columns; (2) the planned `POST /api/detections/:id/correct` is dominated by the existing `PATCH /api/projects/:pid/recognition_items/:id/correct` (in `api/recognitions.rs`) which already does set/clear with audit + UploadPerm + person/tool/device targets; (3) the gallery half of #7 is already implemented in `server/src/finetune.rs` (M2 turn 11) — corrections → `identity_embeddings` rows tagged `source='manual'`. Net: #5-skel shipped with only (i) the worker `target_type='tool'` literal + doc-comment updates and (ii) a new audit table `model_versions` + read-only view `v_training_corrections` to feed #7. #7's remaining work narrows to the detector-retrain half (YOLO retrain → ONNX export → shadow-eval gate → atomic promote). Lesson preserved in long-term notes: API path 先 grep 不要猜 — should have checked `api/recognitions.rs` and `finetune.rs` before drafting #5-skel scope at c853773; cost was one extra round-trip with King but no wasted code.
- **#7 split into #7a / #7b / #7c (decided at 2026-04-28 +08, before #7a implementation).** The remaining detector-retrain work has three axes with very different blast-radius and runtime costs, so they ship as three sub-milestones rather than one big bang: **#7a** is data prep only (pure Rust + DB read, no GPU, no shell-outs, no DB writes; exercisable via `--dry-run` even with zero corrections in the DB). **#7b** is the ultralytics train + ONNX export (~30 min CPU; the only step that needs the PyTorch runtime). **#7c** is the gate + atomic promote + `model_versions` audit row (the only step that mutates production-visible state — `models/object_detect.onnx` and the `model_versions` table). The split lets the bbox→YOLO conversion + cycle directory layout get reviewed and exercised independently of the long train, and gives operators a `--dry-run` they can use today to inspect the eligible-corrections funnel. Single-class detector (id 0 = `tool`) is locked in per the #5-skel scope revision (`device` is folded into `tool` for new writes).
- **#7b further split into #7b-prep + #7b-train (discovered while scoping #7b implementation, 2026-04-28 +08).** Inspecting `server/src/inference/yolov8.rs` before shelling out to ultralytics revealed that `NUM_CHANNELS=84` / `NUM_CLASSES=80` were `pub const` and the decoder asserted `out0.len() == NUM_CHANNELS * NUM_ANCHORS`. A single-class retrained candidate produces `[1, 5, 8400]` and would have panicked at runtime — i.e. #7b in its original shape would have produced an ONNX file the server could not load. Rather than ship #7b and then a follow-up hot-fix, the inference-decoder fix was carved out as **#7b-prep** (pure Rust, zero ultralytics dependency, runs in seconds with full test coverage) and the ultralytics shell-out + ONNX export is now **#7b-train**. The lesson preserved in long-term notes: before adding a new model variant, grep the inference path for hard-coded shape constants — `NUM_CHANNELS` was the obvious blocker, but the same class of issue (hard-coded opset, hard-coded input dims, hard-coded number of detection heads) can hide elsewhere. #7b-prep also incidentally documents the legacy COCO shape via `LEGACY_COCO_NUM_CHANNELS` / `LEGACY_COCO_NUM_CLASSES` so future readers can see the historical default without spelunking git.
